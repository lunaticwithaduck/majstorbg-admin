# Migration Patterns

## When to Migrate

### Decision Framework

| Signal | Recommendation |
|--------|---------------|
| Team unfamiliar with Redux patterns, high onboarding cost | Consider Zustand (simpler mental model) |
| Heavy boilerplate (action type constants, switch reducers, sagas) | Migrate to Redux Toolkit at minimum |
| Only using Redux for API data caching | Replace with RTK Query or TanStack Query |
| Want to remove Redux entirely | Zustand (client state) + TanStack Query (server state) |
| App works fine, no developer pain | Don't migrate — migration has real cost |
| Starting a new feature module | Use new library for new code, leave existing alone |

### Migration Effort

| From | To | Effort | Risk | Incremental? |
|------|----|--------|------|-------------|
| Legacy Redux → | Redux Toolkit (createSlice) | Low | Low | Yes — one slice at a time |
| Redux + Saga → | RTK Query | Medium | Medium | Yes — one API at a time |
| Redux + Thunk → | RTK Query | Low-Medium | Low | Yes — one API at a time |
| RTK Query → | TanStack Query | Medium | Medium | Yes — one endpoint at a time |
| Redux (any) → | Zustand | Low | Low | Yes — one store/slice at a time |
| Redux (any) → | Zustand + TanStack Query | Medium-High | Medium | Yes — split client vs server state |

## Legacy Redux → Redux Toolkit

### Before: Traditional Redux

```tsx
// constants.ts
const ADD_TODO = 'todos/ADD_TODO'
const TOGGLE_TODO = 'todos/TOGGLE_TODO'
const REMOVE_TODO = 'todos/REMOVE_TODO'

// actions.ts
const addTodo = (title: string) => ({
  type: ADD_TODO,
  payload: { id: Date.now().toString(), title, completed: false },
})
const toggleTodo = (id: string) => ({ type: TOGGLE_TODO, payload: id })
const removeTodo = (id: string) => ({ type: REMOVE_TODO, payload: id })

// reducer.ts
function todosReducer(state: Todo[] = [], action: any): Todo[] {
  switch (action.type) {
    case ADD_TODO:
      return [...state, action.payload]
    case TOGGLE_TODO:
      return state.map(t => t.id === action.payload ? { ...t, completed: !t.completed } : t)
    case REMOVE_TODO:
      return state.filter(t => t.id !== action.payload)
    default:
      return state
  }
}
```

### After: Redux Toolkit

```tsx
// todosSlice.ts — replaces constants + actions + reducer
import { createSlice, type PayloadAction } from '@reduxjs/toolkit'

const todosSlice = createSlice({
  name: 'todos',
  initialState: [] as Todo[],
  reducers: {
    addTodo: {
      reducer(state, action: PayloadAction<Todo>) {
        state.push(action.payload) // Immer allows "mutation"
      },
      prepare(title: string) {
        return { payload: { id: Date.now().toString(), title, completed: false } }
      },
    },
    toggleTodo(state, action: PayloadAction<string>) {
      const todo = state.find(t => t.id === action.payload)
      if (todo) todo.completed = !todo.completed
    },
    removeTodo(state, action: PayloadAction<string>) {
      return state.filter(t => t.id !== action.payload)
    },
  },
})

export const { addTodo, toggleTodo, removeTodo } = todosSlice.actions
export default todosSlice.reducer
```

### Step-by-Step Migration

1. Install `@reduxjs/toolkit` (already includes Redux, Immer, Reselect)
2. Replace `createStore` with `configureStore` (drop-in, adds middleware + devtools)
3. Convert one reducer at a time: `switch/case` → `createSlice`
4. Delete the constants file (action types are auto-generated)
5. Delete the action creators file (auto-generated by createSlice)
6. Update imports in components
7. Repeat for each slice

**Key:** Old reducers and new slices can coexist in `configureStore.reducer`.

## Redux + Saga → RTK Query

### Before: Redux Saga for API

```tsx
// sagas/usersSaga.ts
function* fetchUsersSaga() {
  try {
    yield put({ type: 'FETCH_USERS_PENDING' })
    const response: Response = yield call(fetch, '/api/users')
    const data: User[] = yield call([response, 'json'])
    yield put({ type: 'FETCH_USERS_FULFILLED', payload: data })
  } catch (error) {
    yield put({ type: 'FETCH_USERS_REJECTED', payload: error.message })
  }
}

function* watchFetchUsers() {
  yield takeLatest('FETCH_USERS_REQUEST', fetchUsersSaga)
}
```

### After: RTK Query

```tsx
// api/usersApi.ts
import { api } from '@/lib/api'

const usersApi = api.injectEndpoints({
  endpoints: (builder) => ({
    getUsers: builder.query<User[], void>({
      query: () => '/users',
      providesTags: ['User'],
    }),
  }),
})

export const { useGetUsersQuery } = usersApi

// Component — no dispatch, no action, no selector
function UserList() {
  const { data: users, isLoading, error } = useGetUsersQuery()
  // ...
}
```

### Migration Steps

1. Set up RTK Query `createApi` alongside existing saga setup
2. Pick one API endpoint to migrate (start with read-only queries)
3. Create the RTK Query endpoint
4. Update the component to use the generated hook
5. Remove the saga watcher, action types, and reducer cases for that endpoint
6. Repeat for each API endpoint
7. When all sagas are migrated, remove `redux-saga` dependency

## RTK Query → TanStack Query

### Why Migrate

- Removing Redux from the app entirely
- Prefer TanStack Query's devtools and cache management
- Moving to a non-Redux state management approach

### Endpoint Mapping

| RTK Query | TanStack Query |
|-----------|---------------|
| `builder.query` | `useQuery` |
| `builder.mutation` | `useMutation` |
| `providesTags` | Query key structure |
| `invalidatesTags` | `invalidateQueries({ queryKey })` |
| `pollingInterval` | `refetchInterval` |
| `skip` | `enabled: false` |
| `transformResponse` | `select` option or transform in `queryFn` |
| `onQueryStarted` (optimistic) | `onMutate` / `onError` / `onSettled` |

### Before: RTK Query

```tsx
const api = createApi({
  endpoints: (builder) => ({
    getUsers: builder.query<User[], void>({
      query: () => '/users',
      providesTags: ['User'],
    }),
    createUser: builder.mutation<User, CreateUserInput>({
      query: (body) => ({ url: '/users', method: 'POST', body }),
      invalidatesTags: ['User'],
    }),
  }),
})

const { useGetUsersQuery, useCreateUserMutation } = api
```

### After: TanStack Query

```tsx
const userKeys = {
  all: ['users'] as const,
  lists: () => [...userKeys.all, 'list'] as const,
  detail: (id: string) => [...userKeys.all, 'detail', id] as const,
}

function useUsers() {
  return useQuery({
    queryKey: userKeys.lists(),
    queryFn: () => fetch('/api/users').then(r => r.json()),
  })
}

function useCreateUser() {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: (data: CreateUserInput) =>
      fetch('/api/users', { method: 'POST', body: JSON.stringify(data) }).then(r => r.json()),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: userKeys.lists() })
    },
  })
}
```

## Redux → Zustand (Client State)

### Before: Redux Slice

```tsx
const uiSlice = createSlice({
  name: 'ui',
  initialState: { sidebarOpen: false, theme: 'light' as 'light' | 'dark' },
  reducers: {
    toggleSidebar(state) { state.sidebarOpen = !state.sidebarOpen },
    setTheme(state, action: PayloadAction<'light' | 'dark'>) { state.theme = action.payload },
  },
})

// Component
const sidebarOpen = useAppSelector(state => state.ui.sidebarOpen)
const dispatch = useAppDispatch()
dispatch(toggleSidebar())
```

### After: Zustand Store

```tsx
import { create } from 'zustand'

interface UIStore {
  sidebarOpen: boolean
  theme: 'light' | 'dark'
  toggleSidebar: () => void
  setTheme: (theme: 'light' | 'dark') => void
}

const useUIStore = create<UIStore>((set) => ({
  sidebarOpen: false,
  theme: 'light',
  toggleSidebar: () => set((s) => ({ sidebarOpen: !s.sidebarOpen })),
  setTheme: (theme) => set({ theme }),
}))

// Component — no dispatch, no selector function
const sidebarOpen = useUIStore((s) => s.sidebarOpen)
const toggleSidebar = useUIStore((s) => s.toggleSidebar)
toggleSidebar()
```

### Incremental Approach

1. Create Zustand stores for UI state first (sidebar, modals, theme)
2. Update components to use Zustand hooks instead of `useAppSelector`/`dispatch`
3. Remove the corresponding Redux slice
4. Continue with other client-state slices
5. Keep Redux only for slices that need RTK Query integration

## Bridge Pattern (Coexistence)

When migrating incrementally, both stores coexist:

```tsx
// Both providers wrap the app
<Provider store={reduxStore}>
  {/* Zustand needs no provider — stores are module-level */}
  <QueryClientProvider client={queryClient}>
    <App />
  </QueryClientProvider>
</Provider>
```

### Reading Redux State from Zustand (Bridge)

```tsx
// Temporary bridge during migration
import { store as reduxStore } from '@/lib/reduxStore'

const useBridgeStore = create<BridgeState>((set) => {
  // Subscribe to Redux changes
  reduxStore.subscribe(() => {
    const state = reduxStore.getState()
    set({ user: state.auth.user })
  })

  return {
    user: reduxStore.getState().auth.user,
  }
})
```

Remove the bridge once the auth slice is fully migrated to Zustand.

### File-by-File Component Migration

```tsx
// Before (Redux):
import { useAppSelector, useAppDispatch } from '@/lib/hooks'
import { toggleSidebar } from '@/lib/slices/uiSlice'

// After (Zustand):
import { useUIStore } from '@/lib/stores/uiStore'

// Only the import changes — component JSX stays the same
```
